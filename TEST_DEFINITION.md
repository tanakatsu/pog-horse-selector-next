# POG馬選択Webアプリ クローン - テスト計画書

## 1. テスト方針

### テスト種別と対象

| 種別 | ツール | 対象 |
|-----|--------|------|
| 単体テスト | Vitest + React Testing Library | バリデーションロジック、ユーティリティ関数、Zustandストア |
| コンポーネントテスト | Vitest + React Testing Library | UIコンポーネントの表示・インタラクション |
| E2Eテスト | Playwright | 認証フロー、主要ユーザーシナリオ |

### テスト環境

- **単体・コンポーネントテスト**: ローカル（CI含む）。Supabaseはモック化
- **E2Eテスト**: Supabase Test環境（専用プロジェクト or ローカルSupabase）

## 2. 単体テスト

### 2-1. バリデーション（`src/lib/validations/`）

#### `horse.ts` - 馬フォームバリデーション

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: 有効なhorse_id | `"2025105006"` | バリデーション通過 |
| 異常系: horse_idが9桁 | `"202510500"` | エラー |
| 異常系: horse_idが11桁 | `"20251050060"` | エラー |
| 異常系: horse_idに数字以外 | `"2025ABCDEF"` | エラー |
| 異常系: 馬名が1文字 | `"ア"` | エラー（2文字以上必須） |
| 正常系: 馬名が2文字 | `"アア"` | バリデーション通過 |
| 異常系: 馬名が既存と重複 | 既存: `["キズナの2025"]`, 入力: `"キズナの2025"` | エラー（重複） |
| 正常系: 編集時に自身の馬名 | 編集対象と同じ名前 | バリデーション通過（自身は除外） |
| 異常系: 母馬名が既存と重複 | 既存: `["キズナ"]`, 入力: `"キズナ"` | エラー（重複） |
| 正常系: 編集時に自身の母馬名 | 編集対象と同じ母名 | バリデーション通過（自身は除外） |
| 異常系: 父馬名が1文字 | `"ア"` | エラー（2文字以上必須） |
| 異常系: po_nameが空 | `""` | エラー（必須） |

#### `owner.ts` - オーナーフォームバリデーション

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: 有効なオーナー名 | `"山田太郎"` | バリデーション通過 |
| 異常系: オーナー名が空 | `""` | エラー（必須） |

#### `auth.ts` - 認証フォームバリデーション

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: 有効なメール | `"test@example.com"` | バリデーション通過 |
| 異常系: 無効なメール形式 | `"notanemail"` | エラー |
| 異常系: パスワードが短い | `"pass"` | エラー（6文字以上等） |
| 正常系: 有効なパスワード | `"password123"` | バリデーション通過 |

## 5. テスト実行方法

### 単体・コンポーネントテスト

```bash
# テスト実行
npm run test

# ウォッチモード
npm run test:watch

# カバレッジ
npm run test:coverage
```

### E2Eテスト

```bash
# ローカルE2E（事前に開発サーバーを起動）
npm run dev &
npx playwright test

# UIモードで実行（デバッグ用）
npx playwright test --ui

# 特定のテストのみ実行
npx playwright test e2e/auth.spec.ts
```

### E2Eテスト用の環境変数（`.env.test`）

```
NEXT_PUBLIC_SUPABASE_URL=<test用supabaseのURL>
NEXT_PUBLIC_SUPABASE_ANON_KEY=<test用supabaseのANON_KEY>
NEXT_PUBLIC_TARGET_YEAR=2025
TEST_USER_EMAIL=test@example.com
TEST_USER_PASSWORD=testpassword123
```

## 7. テスト優先度

| 優先度 | 対象 | 理由 |
|-------|------|------|
| 高 | バリデーションロジック（horse.ts） | ビジネスルールの中核。重複チェック・ID形式 |
| 高 | E2E: 馬登録フロー (TC-HOME-003〜008) | 最も利用頻度が高く、バグの影響が大きい |
| 高 | E2E: 認証フロー (TC-AUTH-001〜006) | 全機能の前提 |
| 高 | E2E: RLS/データ分離 (TC-SEC-001〜002) | セキュリティ要件 |
| 中 | E2E: オーナー管理 (TC-GROUP-001〜005) | 馬登録の前提となる機能 |
| 中 | Zustandストア selectors | データ表示ロジックの正確性 |
| 中 | E2E: CSVエクスポート (TC-DOWNLOAD-002) | 出力データの正確性 |
| 低 | コンポーネントテスト全般 | E2Eでカバーされる部分が多い |
